# Dify 工作流说明

本目录用于保存项目对应的 Dify Chatflow 导出文件，方便直接导入 Dify 平台并复现实验环境。

## 文件说明

### 1. `电商全能客服.yml`

原始版本工作流，保留了最初的业务分支结构，适合作为对照版本查看。

主要特点：

- 包含售前 / 售后 / 订单 / 闲聊等分支
- 使用问题分类器进行场景划分
- 使用知识检索节点进行业务问答增强
- 末端存在变量聚合后统一回复的结构

### 2. `电商全能客服-流式优化版.yml`

流式优化后的工作流版本，是更推荐导入和演示的版本。

主要优化点：

- 移除了“变量聚合器 + 统一 `Answer`”结构
- 改为每个业务分支直接连接自己的 `Answer`
- 在耗时分支前增加前置提示 `Answer`
- 更适合在 Dify Chatflow 中实现渐进式输出

## 推荐导入版本

如果你希望在本项目中获得更好的流式交互体验，建议优先导入：

- `电商全能客服-流式优化版.yml`

## 导入步骤

1. 打开 Dify 控制台
2. 创建或进入一个 Chatflow 应用
3. 选择导入 DSL / YAML 文件
4. 导入本目录中的目标工作流文件
5. 检查模型、知识库和代码节点配置
6. 发布应用并生成新的 App API Key
7. 将该 Key 填入项目根目录 `.env` 中的 `DIFY_API_KEY`

## 导入后需要检查的内容

### 1. 模型供应商

工作流中默认使用的是阿里通义系列模型。如果你的 Dify 环境没有完全相同的模型：

- 可替换为同能力的聊天模型
- 需要确保分类器、LLM 节点都重新绑定可用模型

### 2. 知识库绑定

导出的工作流中知识检索节点引用的是原始环境中的知识库 ID。导入到新环境后，通常需要：

- 重新绑定你自己的知识库
- 确保售前 / 售后相关文档已经导入

### 3. Code 节点

订单查询分支中存在 `Code` 节点模拟查库逻辑。导入后需要确认：

- Dify Sandbox 服务可用
- 如果是自建 Dify，Sandbox、证书和反代配置正常
- 如果不想依赖代码沙箱，也可以将订单查询逻辑改成 LLM 模拟查单

## 与项目代码的关系

本项目后端只负责：

- 将用户消息发送给 Dify
- 接收并转发 SSE 事件
- 保存会话和消息历史

真正决定“回复内容”和“业务分支行为”的，是你在 Dify 中发布的工作流。因此：

- 更换工作流后，需要重新生成并替换 App API Key
- 如果工作流结构变化较大，建议先在 Dify 平台中单独调试通过，再接入本项目

## 建议的演示方式

如果你用于课程设计、作品集或答辩展示，建议按下面顺序演示：

1. 展示工作流全图
2. 说明问题分类、知识检索、订单分支和闲聊分支
3. 强调流式优化版通过多 `Answer` 实现渐进式输出
4. 回到前端页面演示真实聊天效果
5. 展示数据库中的历史消息与会话记录

这样能同时体现：

- AI 工作流编排能力
- 全栈工程实现能力
- 真实联调与问题排查能力
